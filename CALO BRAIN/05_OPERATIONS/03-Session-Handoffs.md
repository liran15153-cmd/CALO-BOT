# Development Log

## 2026-06-20

- Added the v1 workout execution loop: current plans now expose row-level workout/exercise ids, `/api/workouts/next` selects the next plan workout from recent logs, structured `POST /api/workout-logs` persists exercise/set/RPE/RIR/pain details in `WorkoutLog.exercise_results`, and the Workouts UI shows "האימון הבא" with a structured logging form while keeping free-text logging.
- Expanded `TrainingAdaptationService` to return `load_signal`, `next_adjustment`, and `exercise_adjustments` for progression, high-RPE recovery, pain caution, and skipped/partial minimum-version cases. Also tightened extreme-diet safety so ordinary meal calorie mentions are not blocked unless they appear in a daily restriction or rapid-loss context.
- Connected the dashboard next action to the same plan-backed next-workout service: `/api/dashboard` now includes a compact `next_workout`, completed logs advance the recommendation, skipped/partial/pain logs keep the same workout with a conservative adjustment, and the frontend renders natural Hebrew labels instead of internal `load_signal` codes.
- Added a derived `execution_plan` to `/api/workouts/next` so adaptation becomes executable: skipped/partial logs return a shorter minimum version, pain logs reduce/swap conservatively, high-RPE logs reduce or hold, and progress candidates get only a small one-variable progression cue while the saved base plan remains unchanged.
- Added `GET /api/workout-logs/recent` and a recent workout-log history in the Workouts UI so saved free-text/structured logs remain visible after refresh, with Hebrew status/confidence labels, dates, RPE, pain flags, notes, and exercise summaries.
- Added `GET /api/meals/recent` and a recent-meals section in the Meals UI so manual/image meal logs become a visible history with approximate calorie/protein ranges, confidence labels, and detected/manual items instead of disappearing after submission.
- Added a secondary dashboard "תזונה היום" action using today's meal/workout state: no meals prompts one approximate meal log, post-workout gaps suggest a protein anchor, and logged meals show a simple continue-tracking cue with today's protein range when available.
- Added `GET /api/summaries/weekly/current` and a compact weekly review in the Dashboard UI. The new endpoint previews the current week from stored logs/meals without creating a `WeeklySummary` row or incrementing summary usage, while explicit weekly-summary requests still persist and track usage.
- Fixed dashboard streak semantics: `current_streak` now counts consecutive active dates from completed/partial/modified workouts and meal logs instead of reusing the number of completed workout logs; skipped workouts alone no longer inflate the streak.
- Made dashboard goals more state-aware: when no profile exists but an active workout plan does, `current_goal` now falls back to the saved plan goal instead of showing an empty-goal state.
- Started migrating open knowledge questions from canned local intents to provider + retrieval, gradually and non-regressively. `fitness_term_guidance`, `nutrition_guidance`, `equipment_substitution_guidance`, `missed_workout_guidance` and `weekly_action_plan_guidance` now route through the AI provider with retrieved coaching knowledge when `ANTHROPIC_API_KEY` is set; the existing canned answer is kept as a fallback when the provider is unavailable, returns non-Hebrew, errors, or the daily budget is spent. Safety-flavored intents (supplement/creatine/knee/low-energy) and closed actions (logging, plan/summary generation) stay local, and `SafetyService` still runs first.
- Mapped each migrated intent to the context family that retrieves the most relevant knowledge (nutrition→meal, equipment/weekly-plan→workout_plan, missed-workout→workout_log) and added `_respond_local_tool` plus `_KNOWLEDGE_INTENT_FALLBACKS`/`_PROVIDER_CONTEXT_INTENT_FOR` in `coach_engine.py`. Behavior is unchanged without an API key, so all existing tests stay green; added five tests for the configured-provider, bad-Hebrew-fallback, meal-context, kept-local, and budget-fallback paths. Verified live: "מה זה דילואד" now answers from the provider grounded in deload knowledge instead of canned text, while creatine stays local and chest-pain still hits safety.
- Closed the coaching-knowledge runtime gap systematically: message retrieval now discovers every knowledge table, single dict and rule list in `_BASE_CONTEXT` automatically (`_iter_knowledge_entries`) instead of a hand-maintained 15-table allow-list, raising reachable top-level keys from 31/78 to 74/78 (only pure-metadata keys are skipped).
- Made `_compact_knowledge_hit` project any entry shape into a compact hit via broad guidance/action/avoid field buckets plus a generic fallback, so previously-invisible tables (exercise_library, goal_specific_programming, technique_cues, anatomy_muscle_map, cardio/running, etc.) surface real content.
- Added Hebrew prefix-aware tokenization (`_token_variants`) so attached prefixes (ב/ה/ו/כ/ל/מ/ש) no longer block matching, e.g. "בחלבון" now matches "חלבון".
- Added targeted topic aliases for high-value clusters that missed on transliteration/morphology: protein, deload, running, hypertrophy, strength, zone 2, plateau, squat, plate building.
- Verified end-to-end against the live local bot: protein/running/plateau questions return provider answers grounded in the retrieved knowledge (e.g. the 1.4-2.0 g/kg protein range), and added retrieval-coverage tests. Known residual: questions routed to local hardcoded intents (e.g. deload, hypertrophy) still answer with canned text and bypass retrieval.
- Improved CALO Coach Hebrew style after localhost QA and terminology research: prompts and provider context now require natural Israeli fitness Hebrew, allow useful terms such as RPE/RIR/DOMS/full-body/דילואד/progressive overload, and explicitly avoid literal translations such as מערכות, הישנויות, and פריקת עומס.
- Added a local `fitness_term_guidance` chat intent for common Hebrew coaching questions about RPE/RIR, hypertrophy, progression, deload, and DOMS after provider QA still produced machine-translated phrasing.
- Expanded local Hebrew guidance for Zone 2, full-body vs push/pull/legs, equipment substitutions, missed-workout recovery, and mixed RPE/RIR prompts so stated values like `RIR 3` are preserved instead of overwritten by canned examples.
- Added local warmup/cooldown guidance so common Hebrew questions about חימום, קירור, מתיחות, DOMS, and סט הכנה no longer fall through to generic provider chat.
- Added local low-energy one-action guidance so "אין לי כוח היום" requests get one small practical action instead of provider fallback.
- Added local Hebrew nutrition guidance for workout-adjacent meal choices and food-image calorie uncertainty after live QA showed provider responses with broken Hebrew and irrelevant context.
- Tightened the Hebrew language guard after localhost QA so generic English headings/phrases such as `Weekly summary`, `Action plan`, `recover tomorrow`, and `protein timing` are rejected, while professional terms such as RPE/RIR/DOMS/HIIT/Zone 2/full-body remain allowed.
- Tightened explicit neutral-address handling after localhost QA found provider answers using `אתה`/`הוסף` despite "בלי לפנות בלשון זכר או נקבה"; configured provider text that violates that request is now rejected, knowledge intents fall back locally when possible, and progression/RPE/DOMS/deload local answers use neutral Hebrew phrasing such as `להוסיף`, `לשמור`, and `כדאי`.
- Added local `supplement_safety_guidance` for high-caffeine pre-workout, yohimbine, and fat-burner questions so stimulant-risk prompts no longer fall into meal-timing nutrition guidance.
- Added local `weekly_action_plan_guidance` for short weekly strength-plus-walking plans after provider chat produced literal phrases such as "דחיפת אדמה", "משיכת גוף", and "בהירוג עבור התאוששות".
- Fixed dashboard and chat UI Hebrew QA issues found in Playwright: singular dashboard counts now render "יום אחד" / "אימון אחד שפוספס", and chat provider-status labels no longer stick to the end of the coach response.
- Split dashboard nutrition and missed-workout metrics so the nutrition card no longer describes workout adherence.
- Moved DNP handling ahead of generic rapid-weight-loss safety classification so DNP requests record `dangerous_substance` and return substance-specific conservative guidance.
- Cleaned local workout-plan chat naming so saved-plan responses no longer leak internal split ids like `full_body` and no longer say "1 ימים בשבוע" for one-off workouts.
- Fixed single-session workout-plan inference so Hebrew prompts like a 30-minute gym workout use the requested duration and gym equipment before falling back to profile defaults.
- Added explicit neutral-address Hebrew guidance so local coaching answers, provider prompts, and structured workout-plan guidance avoid direct masculine/feminine address when the user requests neutral wording.
- Fixed workout-plan UI singular wording so one-day plans and one-set exercises render as "יום אחד בשבוע" and "סט אחד".
- Added workout-plan UI display for persisted plan type and duration so single-session plans show "אימון יחיד" and the saved duration such as "30 דקות".
- Cleaned smaller Hebrew UI copy issues in meals/settings: uploaded meal notes no longer produce mixed-language grammar like "Lunch נשמרה", and one deleted record renders as "רשומה אחת נמחקה".
- Translated user-facing structured workout-plan guidance fields so `progression_model`, safety notes, exercise progression/regression, and reduced-readiness notes no longer leak English instructions such as "Stop", "Use", "Reduce", or "Do not".
- Fixed Hebrew singular/plural wording in local daily and weekly summaries so one workout or one meal is shown as "אימון אחד" / "ארוחה אחת" rather than "1 אימונים" / "1 ארוחות".
- Expanded provider-response Markdown cleanup to strip common list, numbered-list, blockquote, and simple table syntax before storing/displaying chat text.
- Fixed the coaching knowledge runtime gap: provider context was previously selected only by `intent`, so the current user message could not retrieve specific protocol knowledge from `coaching_knowledge.py`.

- Added the v1 workout execution loop: current plans now expose row-level workout/exercise ids, `/api/workouts/next` selects the next plan workout from recent logs, structured `POST /api/workout-logs` persists exercise/set/RPE/RIR/pain details in `WorkoutLog.exercise_results`, and the Workouts UI shows "האימון הבא" with a structured logging form while keeping free-text logging.
- Expanded `TrainingAdaptationService` to return `load_signal`, `next_adjustment`, and `exercise_adjustments` for progression, high-RPE recovery, pain caution, and skipped/partial minimum-version cases. Also tightened extreme-diet safety so ordinary meal calorie mentions are not blocked unless they appear in a daily restriction or rapid-loss context.
- Connected the dashboard next action to the same plan-backed next-workout service: `/api/dashboard` now includes a compact `next_workout`, completed logs advance the recommendation, skipped/partial/pain logs keep the same workout with a conservative adjustment, and the frontend renders natural Hebrew labels instead of internal `load_signal` codes.
- Added a derived `execution_plan` to `/api/workouts/next` so adaptation becomes executable: skipped/partial logs return a shorter minimum version, pain logs reduce/swap conservatively, high-RPE logs reduce or hold, and progress candidates get only a small one-variable progression cue while the saved base plan remains unchanged.
- Added `GET /api/workout-logs/recent` and a recent workout-log history in the Workouts UI so saved free-text/structured logs remain visible after refresh, with Hebrew status/confidence labels, dates, RPE, pain flags, notes, and exercise summaries.
- Added `GET /api/meals/recent` and a recent-meals section in the Meals UI so manual/image meal logs become a visible history with approximate calorie/protein ranges, confidence labels, and detected/manual items instead of disappearing after submission.
- Added a secondary dashboard "תזונה היום" action using today's meal/workout state: no meals prompts one approximate meal log, post-workout gaps suggest a protein anchor, and logged meals show a simple continue-tracking cue with today's protein range when available.
- Added `GET /api/summaries/weekly/current` and a compact weekly review in the Dashboard UI. The new endpoint previews the current week from stored logs/meals without creating a `WeeklySummary` row or incrementing summary usage, while explicit weekly-summary requests still persist and track usage.
- Fixed dashboard streak semantics: `current_streak` now counts consecutive active dates from completed/partial/modified workouts and meal logs instead of reusing the number of completed workout logs; skipped workouts alone no longer inflate the streak.
- Made dashboard goals more state-aware: when no profile exists but an active workout plan does, `current_goal` now falls back to the saved plan goal instead of showing an empty-goal state.
- Started migrating open knowledge questions from canned local intents to provider + retrieval, gradually and non-regressively. `fitness_term_guidance`, `nutrition_guidance`, `equipment_substitution_guidance`, `missed_workout_guidance` and `weekly_action_plan_guidance` now route through the AI provider with retrieved coaching knowledge when `ANTHROPIC_API_KEY` is set; the existing canned answer is kept as a fallback when the provider is unavailable, returns non-Hebrew, errors, or the daily budget is spent. Safety-flavored intents (supplement/creatine/knee/low-energy) and closed actions (logging, plan/summary generation) stay local, and `SafetyService` still runs first.
- Mapped each migrated intent to the context family that retrieves the most relevant knowledge (nutrition→meal, equipment/weekly-plan→workout_plan, missed-workout→workout_log) and added `_respond_local_tool` plus `_KNOWLEDGE_INTENT_FALLBACKS`/`_PROVIDER_CONTEXT_INTENT_FOR` in `coach_engine.py`. Behavior is unchanged without an API key, so all existing tests stay green; added five tests for the configured-provider, bad-Hebrew-fallback, meal-context, kept-local, and budget-fallback paths. Verified live: "מה זה דילואד" now answers from the provider grounded in deload knowledge instead of canned text, while creatine stays local and chest-pain still hits safety.
- Closed the coaching-knowledge runtime gap systematically: message retrieval now discovers every knowledge table, single dict and rule list in `_BASE_CONTEXT` automatically (`_iter_knowledge_entries`) instead of a hand-maintained 15-table allow-list, raising reachable top-level keys from 31/78 to 74/78 (only pure-metadata keys are skipped).
- Made `_compact_knowledge_hit` project any entry shape into a compact hit via broad guidance/action/avoid field buckets plus a generic fallback, so previously-invisible tables (exercise_library, goal_specific_programming, technique_cues, anatomy_muscle_map, cardio/running, etc.) surface real content.
- Added Hebrew prefix-aware tokenization (`_token_variants`) so attached prefixes (ב/ה/ו/כ/ל/מ/ש) no longer block matching, e.g. "בחלבון" now matches "חלבון".
- Added targeted topic aliases for high-value clusters that missed on transliteration/morphology: protein, deload, running, hypertrophy, strength, zone 2, plateau, squat, plate building.
- Verified end-to-end against the live local bot: protein/running/plateau questions return provider answers grounded in the retrieved knowledge (e.g. the 1.4-2.0 g/kg protein range), and added retrieval-coverage tests. Known residual: questions routed to local hardcoded intents (e.g. deload, hypertrophy) still answer with canned text and bypass retrieval.
- Improved CALO Coach Hebrew style after localhost QA and terminology research: prompts and provider context now require natural Israeli fitness Hebrew, allow useful terms such as RPE/RIR/DOMS/full-body/דילואד/progressive overload, and explicitly avoid literal translations such as מערכות, הישנויות, and פריקת עומס.
- Added a local `fitness_term_guidance` chat intent for common Hebrew coaching questions about RPE/RIR, hypertrophy, progression, deload, and DOMS after provider QA still produced machine-translated phrasing.
- Expanded local Hebrew guidance for Zone 2, full-body vs push/pull/legs, equipment substitutions, missed-workout recovery, and mixed RPE/RIR prompts so stated values like `RIR 3` are preserved instead of overwritten by canned examples.
- Added local warmup/cooldown guidance so common Hebrew questions about חימום, קירור, מתיחות, DOMS, and סט הכנה no longer fall through to generic provider chat.
- Added local low-energy one-action guidance so "אין לי כוח היום" requests get one small practical action instead of provider fallback.
- Added local Hebrew nutrition guidance for workout-adjacent meal choices and food-image calorie uncertainty after live QA showed provider responses with broken Hebrew and irrelevant context.
- Tightened the Hebrew language guard after localhost QA so generic English headings/phrases such as `Weekly summary`, `Action plan`, `recover tomorrow`, and `protein timing` are rejected, while professional terms such as RPE/RIR/DOMS/HIIT/Zone 2/full-body remain allowed.
- Tightened explicit neutral-address handling after localhost QA found provider answers using `אתה`/`הוסף` despite "בלי לפנות בלשון זכר או נקבה"; configured provider text that violates that request is now rejected, knowledge intents fall back locally when possible, and progression/RPE/DOMS/deload local answers use neutral Hebrew phrasing such as `להוסיף`, `לשמור`, and `כדאי`.
- Added local `supplement_safety_guidance` for high-caffeine pre-workout, yohimbine, and fat-burner questions so stimulant-risk prompts no longer fall into meal-timing nutrition guidance.
- Added local `weekly_action_plan_guidance` for short weekly strength-plus-walking plans after provider chat produced literal phrases such as "דחיפת אדמה", "משיכת גוף", and "בהירוג עבור התאוששות".
- Fixed dashboard and chat UI Hebrew QA issues found in Playwright: singular dashboard counts now render "יום אחד" / "אימון אחד שפוספס", and chat provider-status labels no longer stick to the end of the coach response.
- Split dashboard nutrition and missed-workout metrics so the nutrition card no longer describes workout adherence.
- Moved DNP handling ahead of generic rapid-weight-loss safety classification so DNP requests record `dangerous_substance` and return substance-specific conservative guidance.

- Fixed single-session workout-plan inference so Hebrew prompts like a 30-minute gym workout use the requested duration and gym equipment before falling back to profile defaults.
- Added explicit neutral-address Hebrew guidance so local coaching answers, provider prompts, and structured workout-plan guidance avoid direct masculine/feminine address when the user requests neutral wording.

- Added workout-plan UI display for persisted plan type and duration so single-session plans show "אימון יחיד" and the saved duration such as "30 דקות".
- Cleaned smaller Hebrew UI copy issues in meals/settings: uploaded meal notes no longer produce mixed-language grammar like "Lunch נשמרה", and one deleted record renders as "רשומה אחת נמחקה".
- Translated user-facing structured workout-plan guidance fields so `progression_model`, safety notes, exercise progression/regression, and reduced-readiness notes no longer leak English instructions such as "Stop", "Use", "Reduce", or "Do not".
- Fixed Hebrew singular/plural wording in local daily and weekly summaries so one workout or one meal is shown as "אימון אחד" / "ארוחה אחת" rather than "1 אימונים" / "1 ארוחות".
- Expanded provider-response Markdown cleanup to strip common list, numbered-list, blockquote, and simple table syntax before storing/displaying chat text.

- Added the v1 workout execution loop: current plans now expose row-level workout/exercise ids, `/api/workouts/next` selects the next plan workout from recent logs, structured `POST /api/workout-logs` persists exercise/set/RPE/RIR/pain details in `WorkoutLog.exercise_results`, and the Workouts UI shows "האימון הבא" with a structured logging form while keeping free-text logging.
- Expanded `TrainingAdaptationService` to return `load_signal`, `next_adjustment`, and `exercise_adjustments` for progression, high-RPE recovery, pain caution, and skipped/partial minimum-version cases. Also tightened extreme-diet safety so ordinary meal calorie mentions are not blocked unless they appear in a daily restriction or rapid-loss context.
- Connected the dashboard next action to the same plan-backed next-workout service: `/api/dashboard` now includes a compact `next_workout`, completed logs advance the recommendation, skipped/partial/pain logs keep the same workout with a conservative adjustment, and the frontend renders natural Hebrew labels instead of internal `load_signal` codes.
- Added a derived `execution_plan` to `/api/workouts/next` so adaptation becomes executable: skipped/partial logs return a shorter minimum version, pain logs reduce/swap conservatively, high-RPE logs reduce or hold, and progress candidates get only a small one-variable progression cue while the saved base plan remains unchanged.
- Added `GET /api/workout-logs/recent` and a recent workout-log history in the Workouts UI so saved free-text/structured logs remain visible after refresh, with Hebrew status/confidence labels, dates, RPE, pain flags, notes, and exercise summaries.
- Added `GET /api/meals/recent` and a recent-meals section in the Meals UI so manual/image meal logs become a visible history with approximate calorie/protein ranges, confidence labels, and detected/manual items instead of disappearing after submission.
- Added a secondary dashboard "תזונה היום" action using today's meal/workout state: no meals prompts one approximate meal log, post-workout gaps suggest a protein anchor, and logged meals show a simple continue-tracking cue with today's protein range when available.
- Added `GET /api/summaries/weekly/current` and a compact weekly review in the Dashboard UI. The new endpoint previews the current week from stored logs/meals without creating a `WeeklySummary` row or incrementing summary usage, while explicit weekly-summary requests still persist and track usage.
- Fixed dashboard streak semantics: `current_streak` now counts consecutive active dates from completed/partial/modified workouts and meal logs instead of reusing the number of completed workout logs; skipped workouts alone no longer inflate the streak.
- Made dashboard goals more state-aware: when no profile exists but an active workout plan does, `current_goal` now falls back to the saved plan goal instead of showing an empty-goal state.
- Started migrating open knowledge questions from canned local intents to provider + retrieval, gradually and non-regressively. `fitness_term_guidance`, `nutrition_guidance`, `equipment_substitution_guidance`, `missed_workout_guidance` and `weekly_action_plan_guidance` now route through the AI provider with retrieved coaching knowledge when `ANTHROPIC_API_KEY` is set; the existing canned answer is kept as a fallback when the provider is unavailable, returns non-Hebrew, errors, or the daily budget is spent. Safety-flavored intents (supplement/creatine/knee/low-energy) and closed actions (logging, plan/summary generation) stay local, and `SafetyService` still runs first.
- Mapped each migrated intent to the context family that retrieves the most relevant knowledge (nutrition→meal, equipment/weekly-plan→workout_plan, missed-workout→workout_log) and added `_respond_local_tool` plus `_KNOWLEDGE_INTENT_FALLBACKS`/`_PROVIDER_CONTEXT_INTENT_FOR` in `coach_engine.py`. Behavior is unchanged without an API key, so all existing tests stay green; added five tests for the configured-provider, bad-Hebrew-fallback, meal-context, kept-local, and budget-fallback paths. Verified live: "מה זה דילואד" now answers from the provider grounded in deload knowledge instead of canned text, while creatine stays local and chest-pain still hits safety.
- Closed the coaching-knowledge runtime gap systematically: message retrieval now discovers every knowledge table, single dict and rule list in `_BASE_CONTEXT` automatically (`_iter_knowledge_entries`) instead of a hand-maintained 15-table allow-list, raising reachable top-level keys from 31/78 to 74/78 (only pure-metadata keys are skipped).
- Made `_compact_knowledge_hit` project any entry shape into a compact hit via broad guidance/action/avoid field buckets plus a generic fallback, so previously-invisible tables (exercise_library, goal_specific_programming, technique_cues, anatomy_muscle_map, cardio/running, etc.) surface real content.
- Added Hebrew prefix-aware tokenization (`_token_variants`) so attached prefixes (ב/ה/ו/כ/ל/מ/ש) no longer block matching, e.g. "בחלבון" now matches "חלבון".
- Added targeted topic aliases for high-value clusters that missed on transliteration/morphology: protein, deload, running, hypertrophy, strength, zone 2, plateau, squat, plate building.
- Verified end-to-end against the live local bot: protein/running/plateau questions return provider answers grounded in the retrieved knowledge (e.g. the 1.4-2.0 g/kg protein range), and added retrieval-coverage tests. Known residual: questions routed to local hardcoded intents (e.g. deload, hypertrophy) still answer with canned text and bypass retrieval.
- Improved CALO Coach Hebrew style after localhost QA and terminology research: prompts and provider context now require natural Israeli fitness Hebrew, allow useful terms such as RPE/RIR/DOMS/full-body/דילואד/progressive overload, and explicitly avoid literal translations such as מערכות, הישנויות, and פריקת עומס.
- Added a local `fitness_term_guidance` chat intent for common Hebrew coaching questions about RPE/RIR, hypertrophy, progression, deload, and DOMS after provider QA still produced machine-translated phrasing.
- Expanded local Hebrew guidance for Zone 2, full-body vs push/pull/legs, equipment substitutions, missed-workout recovery, and mixed RPE/RIR prompts so stated values like `RIR 3` are preserved instead of overwritten by canned examples.
- Added local warmup/cooldown guidance so common Hebrew questions about חימום, קירור, מתיחות, DOMS, and סט הכנה no longer fall through to generic provider chat.
- Added local low-energy one-action guidance so "אין לי כוח היום" requests get one small practical action instead of provider fallback.
- Added local Hebrew nutrition guidance for workout-adjacent meal choices and food-image calorie uncertainty after live QA showed provider responses with broken Hebrew and irrelevant context.
- Tightened the Hebrew language guard after localhost QA so generic English headings/phrases such as `Weekly summary`, `Action plan`, `recover tomorrow`, and `protein timing` are rejected, while professional terms such as RPE/RIR/DOMS/HIIT/Zone 2/full-body remain allowed.
- Tightened explicit neutral-address handling after localhost QA found provider answers using `אתה`/`הוסף` despite "בלי לפנות בלשון זכר או נקבה"; configured provider text that violates that request is now rejected, knowledge intents fall back locally when possible, and progression/RPE/DOMS/deload local answers use neutral Hebrew phrasing such as `להוסיף`, `לשמור`, and `כדאי`.
- Added local `supplement_safety_guidance` for high-caffeine pre-workout, yohimbine, and fat-burner questions so stimulant-risk prompts no longer fall into meal-timing nutrition guidance.
- Added local `weekly_action_plan_guidance` for short weekly strength-plus-walking plans after provider chat produced literal phrases such as "דחיפת אדמה", "משיכת גוף", and "בהירוג עבור התאוששות".
- Fixed dashboard and chat UI Hebrew QA issues found in Playwright: singular dashboard counts now render "יום אחד" / "אימון אחד שפוספס", and chat provider-status labels no longer stick to the end of the coach response.
- Split dashboard nutrition and missed-workout metrics so the nutrition card no longer describes workout adherence.
- Moved DNP handling ahead of generic rapid-weight-loss safety classification so DNP requests record `dangerous_substance` and return substance-specific conservative guidance.

- Fixed single-session workout-plan inference so Hebrew prompts like a 30-minute gym workout use the requested duration and gym equipment before falling back to profile defaults.
- Added explicit neutral-address Hebrew guidance so local coaching answers, provider prompts, and structured workout-plan guidance avoid direct masculine/feminine address when the user requests neutral wording.
- Fixed workout-plan UI singular wording so one-day plans and one-set exercises render as "יום אחד בשבוע" and "סט אחד".
- Added workout-plan UI display for persisted plan type and duration so single-session plans show "אימון יחיד" and the saved duration such as "30 דקות".
- Cleaned smaller Hebrew UI copy issues in meals/settings: uploaded meal notes no longer produce mixed-language grammar like "Lunch נשמרה", and one deleted record renders as "רשומה אחת נמחקה".
- Translated user-facing structured workout-plan guidance fields so `progression_model`, safety notes, exercise progression/regression, and reduced-readiness notes no longer leak English instructions such as "Stop", "Use", "Reduce", or "Do not".
- Fixed Hebrew singular/plural wording in local daily and weekly summaries so one workout or one meal is shown as "אימון אחד" / "ארוחה אחת" rather than "1 אימונים" / "1 ארוחות".
- Expanded provider-response Markdown cleanup to strip common list, numbered-list, blockquote, and simple table syntax before storing/displaying chat text.
